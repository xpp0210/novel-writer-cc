# 同步协议

## resync 流程（章节修改后必须执行）

任何导致正文内容变化的操作（修复、手动改写、大纲调整后的重写），完成后必须执行 resync。

### 3 步流程

1. **重新提取 + 结算**：Read 正文 → 提取结构化 JSON → 更新 chapter-index + 台账 + handoff + 角色状态
2. **删除旧审计报告**：内容已变，旧报告引用的行号无效
3. **判断是否重新审计**：大段重写（>30% 内容变动）→ 重新审计；小幅修补（改几句话）→ 跳过

### resync 更新的文件

| 文件 | 更新内容 |
|------|----------|
| chapter-index.json | 字数、摘要、关键事件、角色变化 |
| 伏笔台账.json | 新增/推进/兑现的伏笔 |
| 冲突台账.json | 新增/升级/解决的冲突 |
| character-state.json | 角色状态变化 |
| session-handoff.md | 伏笔+冲突列表、角色状态、最近章节概要 |

## 台账健康检查（resync 后必做）

台账不会自动感知正文变化。只要正文变了，都要检查台账是否同步。

**检查 3 项**：

1. **chapter-index.json**：最新章节的摘要/关键事件是否与当前正文一致（参数数值是否同步）
2. **冲突台账.json**：搜索同角色+同事件的重复条目，合并为一条
3. **伏笔台账.json**：搜索语义相同的重复条目，删除重复保留最早的一条

检查方法：直接读取三个 JSON 文件，比对内容。去重判断需要语义理解。

## session-handoff.md 同步检查

每次写新章前或手动更新 handoff 后都要验证：

1. 伏笔条数 = `伏笔台账.json` 中 status=活跃 的条数
2. 冲突条数 = `冲突台账.json` 中 status=活跃 的条数
3. 概要无重复前缀（如"第3章「第3章「标题」」"是 bug）
4. 概要从 chapter-index.json 的 key_events 同步，用分号分隔
5. 冲突条目如有 expected_resolution，在 handoff 中标注预计解决章节

如果条目数不一致，从台账重建 handoff。

## 台账瘦身（每 5 章执行一次）

当章节号为 5 的倍数时，在结算后自动执行：

1. Read `伏笔台账.json`
2. 遍历所有条目：
   - status=已兑现 → 删除 note 字段
   - status=已推进且距 planted_chapter>10 → 删除 note 字段
   - 其他活跃条目 → note 截断到 80 字
3. Write `伏笔台账.json`
4. 对 `冲突台账.json` 执行相同操作

## 修复原则

- 删除情节 > 改因果链 > 直接修补
- 无出处引言 → 转为主角自己的推测
- 修某个词/数值时，Grep 搜全章所有出现处，不能只改标注的那一行
- 批量改数值/术语后必须全文件 Grep 验证，确认零残留
- 轻量验证：小幅修补后用 Grep 扫禁用词即可，不要重跑完整审计
